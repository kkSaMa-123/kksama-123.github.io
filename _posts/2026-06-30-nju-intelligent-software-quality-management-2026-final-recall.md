## 1. DoD 的作用、举例，以及和用户故事验收标准的区别

考到了 Definition of Done，也就是“完成的定义”。

需要说明 DoD 的作用，例如：

1. 统一团队对“完成”的理解，避免只写完代码就认为完成。
2. 保证每个增量都达到可交付质量。
3. 作为团队质量门禁，推动测试、评审、文档、集成等活动前移。

可以举例：一个用户故事只有在代码完成、单元测试通过、代码评审通过、集成测试通过、无阻塞缺陷、部署到测试环境并更新必要文档之后，才算 Done。

和用户故事验收标准的区别：

1. DoD 是团队层面的通用质量标准，适用于所有用户故事或增量。
2. 验收标准是某个用户故事自己的功能条件，用来判断这个故事是否满足业务需求。
3. DoD 更偏过程和质量，验收标准更偏功能和用户价值。

## 2. WIP 是什么、硬币游戏、Little 定理、最佳 WIP 是几，过大过小会怎样

WIP 是 Work In Progress，即在制品数量，表示系统中正在处理但尚未完成的工作项数量。

硬币游戏一般用来说明批量大小、并行工作、流动效率和反馈速度之间的关系。批量越大，等待越多，交付越慢；小批量可以让工作更快流动，也更早暴露问题。

Little 定理：

```text
WIP = Throughput * Cycle Time
```

也可以理解为：

```text
Cycle Time = WIP / Throughput
```

最佳 WIP 不是越大越好，也不是越小越好，而是接近系统瓶颈可稳定处理的数量。它应该让关键资源不空闲，同时又不造成大量排队。

WIP 过大：

1. 排队和上下文切换增加。
2. 周期时间变长。
3. 问题反馈变慢。
4. 隐藏瓶颈和质量问题。

WIP 过小：

1. 人员或资源可能空闲。
2. 吞吐量不足。
3. 系统无法充分利用能力。

## 3. TDD、简单设计、重构、CI/CD 如何相互作用构成整体，并举例说明缺一个会怎样

这四个实践可以看成一个相互支撑的工程闭环。

TDD 通过先写测试明确需求和行为，提供快速反馈；简单设计要求当前设计只满足当前需要，避免过度设计；重构在测试保护下持续改善代码结构；CI/CD 负责频繁集成、自动测试和持续交付，把个人实践扩展为团队级质量保障。

例子：团队开发一个登录功能时，先用 TDD 写失败测试，再实现最小功能；为了避免复杂架构，先采用简单设计；当重复逻辑出现时，通过重构抽取认证模块；每次提交后由 CI 自动运行测试，并通过 CD 部署到测试环境。

缺一个会怎样：

1. 没有 TDD，重构缺少安全网，缺陷容易后移。
2. 没有简单设计，系统可能一开始就过度复杂，后续修改成本高。
3. 没有重构，代码会在迭代中腐化，技术债持续累积。
4. 没有 CI/CD，反馈变慢，集成风险和发布风险增加。

## 4. 激励手段，以及结合马斯洛需求层次理论说明对团队的影响

可以从马斯洛需求层次理论展开：生理需求、安全需求、社交需求、尊重需求和自我实现需求。

对应到软件团队：

1. 薪酬、福利和合理工作量满足基础需求。
2. 稳定预期、清晰目标、公平制度和心理安全感满足安全需求。
3. 团队协作、结对编程、共同复盘和良好沟通满足社交需求。
4. 认可贡献、公开表扬、技术影响力和参与决策满足尊重需求。
5. 自主权、成长机会、挑战性任务和技术愿景满足自我实现需求。

对团队的影响：如果只靠奖金或惩罚，可能只能短期提升产出；如果能同时提供自主、胜任感、归属感和成长空间，团队更容易形成主动负责、持续改进和追求质量的文化。

## 5. 质量管理和软件项目管理有哪些核心挑战与手段

软件项目管理的核心挑战包括需求不确定、估算困难、进度压力、沟通成本、风险控制和干系人期望管理。

常用手段包括：WBS、估算和计划、风险管理、里程碑跟踪、敏捷迭代、燃尽图、挣值管理、每日站会和复盘。

软件质量管理的核心挑战包括缺陷难以完全避免、质量不可只靠后期测试保证、质量属性之间存在权衡、用户质量观和工程质量观可能不一致。

常用手段包括：评审、测试、TDD、CI、静态分析、质量度量、缺陷管理、根因分析、过程改进和持续反馈。

## 6. 软件质量管理要回答的问题，并举两个例子说明

软件质量管理要回答的核心问题包括：

1. 什么是质量？
2. 如何定义质量目标？
3. 如何度量质量？
4. 如何在过程中保证质量？
5. 如何发现、消除和预防缺陷？
6. 如何在成本、进度和质量之间做权衡？

例子一：如果质量目标是减少线上缺陷，那么团队需要回答缺陷从哪里来、在哪个阶段注入、在哪个阶段移除、评审和测试是否有效，以及是否需要进行根因分析。

例子二：如果质量目标是提升用户满意度，那么不仅要关注缺陷数量，还要关注性能、可用性、可靠性、响应速度和用户体验等质量属性。

## 7. Yield 建模以及如何优化

Yield 可以理解为某阶段缺陷移除的有效性，常用于描述评审、测试等质量活动发现并移除缺陷的能力。

可以围绕缺陷注入和缺陷移除建模：统计每个阶段注入多少缺陷、发现多少缺陷、遗留多少缺陷，再分析不同阶段的缺陷移除率。

优化思路：

1. 把缺陷发现活动前移，例如需求评审、设计评审和代码评审。
2. 提高评审质量，例如使用检查表、控制评审速率、记录缺陷类型。
3. 用历史数据识别高缺陷模块和高风险阶段。
4. 做根因分析，减少同类缺陷再次注入。
5. 结合自动化测试和 CI 提供持续反馈。

## 8. A/FT、PQI 是什么，以及作用

A/FT 通常可以理解为 Appraisal to Failure cost ratio，即评价成本与失败成本的比值。评价成本包括评审、测试、检查等质量保障活动成本；失败成本包括修复缺陷、返工、线上故障和客户问题处理成本。

它的作用是帮助团队判断质量投入是否合理。如果评价投入过低，后期失败成本可能很高；如果评价投入过高，也可能造成不必要成本。

PQI 是 Process Quality Index，即过程质量指标，用于综合评价过程质量状态。它通常会结合设计质量、设计评审、代码评审、代码质量和测试质量等方面，判断过程是否能够产生高质量产品。

它的作用是把质量管理从“最后看测试结果”提前到“过程中看质量状态”。

## 9. Quality Journey 八个阶段

题目要求回忆 Quality Journey 的八个阶段。可以按照课程中“质量管理不断前移、不断预防”的思路理解：

1. 测试发现缺陷。
2. 评审提前发现缺陷。
3. 度量缺陷注入和移除。
4. 管理 Yield 等质量指标。
5. 通过设计提升质量。
6. 通过过程改进稳定质量。
7. 通过根因分析预防缺陷。
8. 面向用户价值和多维质量属性持续优化。

这一题最好以课堂讲义版本为准，上面是按记忆和理解整理的答法。

## 10. LLM 应用中有什么是不可替代人类工程师的，为什么

LLM 可以辅助需求整理、代码生成、测试生成、代码解释、缺陷分析和文档编写，但不能完全替代人类工程师。

不可替代的部分包括：

1. 真实需求理解。工程师需要和用户、客户、业务方沟通，判断真正的问题是什么。
2. 价值判断和取舍。质量、成本、进度、风险、安全、隐私之间经常冲突，需要人负责决策。
3. 架构和长期演进责任。LLM 可以给建议，但系统边界、技术路线和长期维护性需要工程师把关。
4. 验证和责任承担。AI 生成内容可能看似正确但隐藏错误，工程师必须通过测试、评审和部署反馈确认质量。
5. 团队协作和过程改进。软件工程不仅是写代码，还包括沟通、承诺、复盘、改进和组织学习。

所以 LLM 更像增强工具，而不是责任主体。AI 可以降低生成成本，但也会提高验证、集成、质量管理和工程判断的重要性。
